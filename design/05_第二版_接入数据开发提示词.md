# 星座百科 · 第二版开发提示词：接入数据

> 接续《04 Claude Code 开发提示词与工作流》。v1 已在最小样例数据上部署；**v2 的目标**：把已生成的完整内容数据（12 星座基线 + 16 题题库 + schema）真正接进算法跑通端到端，补齐还缺的 `config`（年龄 α / 异变阈值 / 匹配权重 / 12×12 相性）与 `copy`（原型名 / 关键词 / 异变激励语 / 匹配解释），并在真实数据下**重新校准异变率**。

## 依赖顺序（为什么这么排）
算法依赖配置，校准又回头改配置，所以**先地基后界面**：
```
A 数据与配置接入地基   B 内容上界面            C 文案库与联调发布
A0 落地数据文件        B0 i18n + 百科页        C0 copy 文案库 + 接 Provider
A1 内容加载层          B1 测试流程(16题)        C1 端到端联调
A2 生成 config         B2 匹配页(基线+矩阵)     C2 发布 v2
A3 接通 core(去样例)
A4 校验回归
A5 重新校准异变率
```
**三支柱延续**：记忆——给 CLAUDE.md 加"数据只经加载层取用"的不变量；Skill——`calibrate` 是这版主角，`design-to-page` 上界面，`quiz-question` 备用；压缩——一阶段一会话，大原型片段交子代理读。

## A · 数据与配置接入地基

### A0 · 落地数据文件
```
把这 7 个文件放进项目对应路径（已生成）：
content/signs/{profiles,zh,en}.json、content/questions/{items,zh,en}.json、src/core/content.schema.ts。
执行 pnpm add zod。写一个 vitest 用例调用 content.schema 的 validateContent({...六个 JSON})，
确认全部通过 schema、且 zh/en 与核心 id 完全对齐。先不要动其它代码。
```

### A1 · 内容加载层（地基，唯一数据入口）
```
建立 src/content/index.ts 作为全项目唯一的内容数据入口：
- 启动时 import 六个 JSON，调用 validateContent 校验一次（失败即抛错，让 build/test 变红）。
- 用 computeMaxScores、computeNeutral 各算一次并缓存。
- 只暴露强类型 getter：getSignProfiles / getSignProfile(id) / getBaseline(id) / getNeutral /
  getMaxScores / getQuestionItems / getSignContent(id,locale) / getQuestionContent(locale)。
然后在根 CLAUDE.md 增加不变量：
"所有内容/配置数据只能经 src/content/index.ts 取用；组件与 src/core 禁止直接 import content/*.json。"
```
> 记忆：这条不变量要写进 CLAUDE.md，它是 v2 防止数据散乱 import 的关键。

### A2 · 生成配置 config（算法的前置）
```
生成 content/config/weights.json，数值取自 docs/02 第 5 节默认值：
{
  "age": { "breakpoints": [
    {"maxAge":17,"alpha":0.10},{"maxAge":25,"alpha":0.20},{"maxAge":35,"alpha":0.35},
    {"maxAge":45,"alpha":0.50},{"maxAge":200,"alpha":0.65} ] },
  "mutation": { "tauBase":0.40, "beta":0.15, "levels":[0.10,0.20] },
  "match": {
    "base":{"w1":0.6,"w2":0.4,"w3":0.0}, "withIdeal":{"w1":0.5,"w2":0.3,"w3":0.2},
    "mutationMultiplier":0.6,
    "personality":{"wSim":0.5,"wComp":0.5,"simDims":["earth","water","order"],"compDims":["fire","water","expr"]}
  }
}

再用脚本按"星座序号距离"确定性生成 content/config/compat-matrix.json（12×12，可复算）：
顺序 aries..pisces=0..11，d=min(|i−j|,12−|i−j|)：
  d=4(同元素三分相)=0.85；d=2(相容元素六分相)=0.75；d=6(对宫)=0.60；d=3(刑克)=0.45；
  d=0(同星座)=0.70；d=1=0.55；d=5=0.50。
结构 { "aries": { "aries":0.70, "taurus":..., ... }, ... }，并预留 overrides 段供人工微调个别配对。

给 content.schema.ts 补 ConfigSchema 与 CompatMatrixSchema（校验取值域、矩阵对称、id 齐全），
并把二者纳入 validateContent 与 src/content 的 getter（getConfig / getCompat(a,b)）。
```

### A3 · 用真实数据+配置接通 core（去样例化）
```
重读 docs/02 第 5 节。移除 v1 里 core 中任何样例/硬编码的 maxScores、星座基线、neutral、权重，
改为由调用方从 src/content 注入（保持 core 纯函数、零框架依赖，不改算法语义）：
- personality.ts 用 getMaxScores() 归一化；
- age.ts 用 getBaseline(sign)+getNeutral()+config.age 算 B_eff；
- mutation.ts 用 config.mutation 的 tauBase/beta/levels 判异变，输出 isMutated/level/drivers；
- matching.ts 用 getBaseline 作候选画像、getCompat 作基础相性、config.match 作权重，含理想型与异变联动。
改完 pnpm typecheck && pnpm test。
```

### A4 · 校验回归
```
tests/ 增加内容回归：
- computeMaxScores === { fire:17, earth:18, air:17, water:14, expr:11, order:23 }
- computeNeutral ≈ { fire:.53, earth:.49, air:.55, water:.52, expr:.58, order:.53 }（容差 0.01）
- README "全选 a" 示例：personality 打分 === { fire:1.0, earth:0, air:~0.65, water:0.5, expr:1.0, order:0 }
- compat-matrix 对称、取值在 [0,1]、12×12 齐全。
全绿再继续。
```

### A5 · 重新校准异变率（calibrate skill 主场）
```
触发 calibrate skill。真实 12 基线 + 16 题接入后必须重测异变率：
1. 写合成用户生成器（scripts/ 或 tests/）：对每个星座的 baseline 加高斯噪声生成"该星座人群"，
   再混入少量均匀随机的"异类"，共 N≥2000，给每个样本随机分配太阳星座与年龄段。
2. 对每样本算 B_eff(sign,age) 与 D=‖V−B_eff‖/√6，按 τ(age)=tauBase−beta·α 判异变；
   统计总异变率与各年龄段异变率。
3. 若总异变率不在 15%–30%，只调 weights.json 的 tauBase/beta（不改逻辑），重跑至达标，
   并确认"年龄越大异变率越高"的单调趋势。把调参前后对比与最终参数写回 weights.json。
```
> 这是 v2 最该慢下来的一步：真实数据下阈值十有八九要重调。改完让 A4 回归。

双罗盘叠图复用 EnergyCompass，每条推荐给"为什么合"的解释（解释文案在 C0 接入，先用占位）。
```

## C · 文案库与联调发布

### C0 · 生成 copy 文案库 + 接入 Provider
```
生成 content/copy/zh.json 与 en.json（两语言结构一致、按 key 对齐），供 TemplateNarrativeProvider 拼装：
{
  "archetypes": [   // 按用户 top-1/top-2 主导维匹配
    {"id":"fire_expr","primary":"fire","secondary":"expr","name":"炽焰先锋","blurb":"…"},
    {"id":"earth_order","primary":"earth","secondary":"order","name":"磐石筑造者","blurb":"…"}
    // …覆盖六个主导维 × 常见次维，至少 12 条
  ],
  "keywords": { "fire":{"high":["热血","果敢"],"low":["温吞"]}, "...":{} },   // 六维高/低关键词池
  "mutation": { "micro":"…", "notable":"…", "extreme":"…这一切都是你一路经历换来的。" },
  "matchReasons": {
    "sameElement":"{element}象同频，默契天成",
    "complement":"你的「{dimU}」恰好接住对方的「{dimB}」",
    "similar":"你们在「{dim}」上高度一致",
    "ideal":"与你描述的理想型高度吻合"
  }
}
要求：原型名原创、禁用 MBTI 四字母；异变文案要"燃"且温暖；中英各自地道、非机翻。
先只产出 archetypes 与 mutation 的中英草稿让我过目，确认语气后再批量补 keywords/matchReasons。
给 copy 补 schema 校验并纳入 validateContent。最后把 TemplateNarrativeProvider 接到这套 copy，
去掉 B1/B2 的占位 fallback。
```
> 这步含创意文案，务必人工复核——别让模型一次性灌满就提交。

## 速查
- **数据入口唯一**：一切内容/配置经 `src/content/index.ts`；core 纯函数靠注入，绝不直接 import JSON。
- **配置先行**：weights.json + compat-matrix.json 在接通算法前生成（A2），否则 matching/age/mutation 接不上。
- **校准是硬步骤**：真实数据接入后 `calibrate` 重测异变率（A5），只动 weights.json 参数、不动逻辑。
- **回归锁数据**：maxScores `{17,18,17,14,11,23}`、neutral `{.53,.49,.55,.52,.58,.53}` 写进测试防漂移。
- **压缩**：A/B/C 各阶段独立会话；大 `.dc.html` 派子代理读；引用文档路径而非粘贴。
- **创意文案人工复核**：C0 的 archetypes/异变语先看草稿再批量。
